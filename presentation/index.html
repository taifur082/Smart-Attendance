ESP32-C6 • UHF RFID • Flask • PostgreSQL • SMS/WhatsApp

Smart Attendance System

IoT-based attendance tracking with RFID and parent notifications

System Architecture

ESP32-C6 (Scanner)→WiFi→Flask Server→PostgreSQL
↘SMS Gateway (Twilio)
↘WhatsApp (Green API)

Single flow: scan at gate → server → DB → parent notification.

Hardware and Components

• ESP32-C6 development board
• J4210U UHF RFID reader module
• UART (e.g. 57600 baud) between ESP32 and reader
• WiFi access point
• RFID tags (EPC) for each student

Each student has a unique EPC; reader reports EPC, RSSI, antenna.

Project Structure

  • main/ — ESP32-C6 firmware: main.cpp, wifi_manager, http_client, config.h
  • server/ — Flask: app.py, models.py, routes.py, config.py, services/ (SMS, WhatsApp), database/
  • components/UHF-driver/ — UHF RFID driver (driver, 903/910, HAL serial)

Firmware in C/C++ (ESP-IDF); server in Python.

ESP32 Firmware — Startup

  1. NVS init (for WiFi)
  2. WiFi manager init and connect (SSID/password from config.h)
  3. HTTP client init
  4. SNTP init (e.g. pool.ntp.org) — wait for time sync
  5. Open UART to reader, GetSettings/SetSettings (e.g. ScanTime 300 ms, BeepOn)
  6. Enter main scan loop

Ref: main/main.cpp app_main().

ESP32 — Scan Loop

  • Every cycle: optional WiFi reconnect check → Inventory(false) (UHF driver) → get tag count.
  • For each tag: GetResult() → EPC bytes → convert to hex string; read RSSI, antenna.
  • Debounce: same EPC within SCAN_DEBOUNCE_SECONDS (e.g. 5 s) → skip send.
  • If not debounced: build scan_data_t (EPC, timestamp ISO, rssi, antenna) → http_client_send_scan_data() → POST to SERVER_URL + SERVER_ENDPOINT.
  • Delay 3 s (continuous mode), repeat.

Timestamp from SNTP; JSON built in http_client.cpp.

Scan Payload (ESP32 → Server)

Endpoint: POST /api/attendance/scan

Header: Authorization: Bearer <API_KEY>

{"epc": "ABCD...", "timestamp": "2026-02-13T10:30:45", "rssi": 130, "antenna": 1}

API key and URL in main/config.h; server URL in server/.env / config.

Server — Handling a Scan

  1. POST /api/attendance/scan → require_api_key → parse JSON, validate epc.
  2. Look up Student by epc (Student.query.filter_by(epc=epc)).
  3. If no student: still create AttendanceLog, log error, return 404.
  4. If student: check cooldown (same student, notification_sent in last N hours).
  5. Create AttendanceLog (student_id, epc, scan_timestamp, rssi, antenna).
  6. If not in cooldown: format message (“Your son/daughter <name> entered at <time>”), then send notification.
  7. Update log (notification_sent, notification_method), commit, return 200.

Ref: server/routes.py scan_attendance().

Parent Notification Logic

  • Message: “Your son/daughter <name> entered the school at <time>.” (SMSService.format_message; son/daughter by gender).
  • Prefer WhatsApp if configured: WhatsAppService.send_message(parent_whatsapp, message) (Green API).
  • If WhatsApp fails and parent_phone set: fallback SMSService.send_sms(parent_phone, message) (Twilio).
  • Cooldown prevents duplicate notifications for same student within N hours.

Credentials in server/.env (Green API, Twilio).

Database Models (PostgreSQL)

Student: id, epc (unique), student_name, parent_name, parent_phone, parent_whatsapp, gender, created_at, updated_at.
AttendanceLog: id, student_id (FK), epc, scan_timestamp, rssi, antenna, notification_sent, notification_method, notification_error, created_at.

ORM: SQLAlchemy; init: python database/init_db.py [--seed].

API Endpoints

  • GET /api/health — no auth; server health.
  • POST /api/attendance/scan — Bearer API key; receive scan (JSON above).
  • GET /api/students — Bearer; list students.
  • POST /api/students — Bearer; create student (epc, student_name, parent_phone, optional parent_name, parent_whatsapp, gender).
  • GET /api/attendance/logs — Bearer; list logs (optional: limit, offset, epc).

All except /api/health require Authorization: Bearer <API_KEY>.

End-to-End User Flow

Student→Reader→ESP32→WiFi→Flask→DB + Notify→Parent
  1. Admin adds students (POST /api/students) with EPC and parent contact.
  2. Student wears/brings RFID tag; passes by reader at school.
  3. ESP32 scans tag → debounce → POST scan to server.
  4. Server finds student, logs attendance, sends WhatsApp or SMS to parent.
  5. Parent receives: “Your son/daughter <name> entered the school at <time>.”

Timeline: student → reader → ESP32 → cloud → parent.

Configuration Summary

ESP32 (main/config.h): WIFI_SSID, WIFI_PASSWORD, SERVER_URL, SERVER_ENDPOINT, API_KEY, SCAN_DEBOUNCE_SECONDS.
Server (.env): DATABASE_URL, API_KEY, Twilio (SID, token, from number), Green API (instance id, token); optional NOTIFICATION_COOLDOWN_HOURS, PREFER_WHATSAPP.

API_KEY must match on ESP32 and server.

Features

Firmware: WiFi + retry, HTTP client + retry, EPC debounce, SNTP time, continuous scanning.
Server: PostgreSQL + SQLAlchemy, REST API, API key auth, SMS (Twilio), WhatsApp (Green API), cooldown, WhatsApp→SMS fallback, error handling and logging.

Security: use HTTPS and strong API keys in production.

Summary

Smart Attendance = RFID scan at gate → ESP32 → Flask → DB + parent notification (WhatsApp/SMS).

Hardware: ESP32 + UHF reader. Firmware: scan + debounce + HTTP POST. Server: lookup + log + notify.
